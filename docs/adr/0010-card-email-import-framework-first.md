# クレジットカードIMAP取込：フレームワーク先行実装と関連設計判断

実クレジットカード各社の利用速報メールサンプルが未入手のため（要件定義書9章）、Phase 4では「送信元メールアドレス→カード会社」「カード会社別パーサー」の具体的な中身を確定できない。そこで以下の3点を、実サンプル入手後に無理なく差し替え・追加できる形にした上で、取込パイプライン自体は今回のフェーズで完成させる方針とした。

## 1. 送信元アドレス・カード下4桁はDB列として持つ（Pythonコード内マッピングにしない）

`m_institutions.card_alert_sender_email`（Fromヘッダのマッチング用）と`m_accounts.card_last4`（下4桁整合性検証用）をnullable列として追加した。ADR 0009で確定した「マスタデータはすべて`m_`テーブルに集約する」方針を踏襲し、Pythonコード内にハードコードした辞書（例: `{"a@b.com": "楽天カード"}`）は持たない。実サンプル入手後は、対象口座のレコードにUPDATE文を1回発行するだけで運用開始でき、コード変更・デプロイは不要。

## 2. カード会社別パーサーは未実装のまま、レジストリ機構のみ先行実装する

`app/services/card_parsers/registry.py`に`institution_name → パーサー実装`のレジストリを用意し、`register_parser()`で登録する設計とした。今回のフェーズではどの会社のパーサーも登録しない（レジストリは空）。送信元アドレスが`m_institutions`で解決できてもパーサー未登録の場合、そのメールは取り込まず「パーサー未実装」件数としてバッチサマリーに計上し、処理全体は継続する。実メールサンプルが入手できた会社から順次`CardEmailParser`実装を追加し登録する（要件定義書4.1章の方針通り）。

## 3. IMAPの重複防止は「既読フラグ」ではなく日付範囲＋source_hashに委ねる

同一メールの再取込を防ぐ手段として、IMAPの`\Seen`フラグ管理（既読化・UNSEEN検索）は採用しない。理由は、既読状態はメールボックス側の状態であり、バッチの再実行やエラー時のリトライで「既読済みだがDB未反映」という不整合が起きやすいため。代わりに、日次バッチは常に「直近N日（既定3日）以内に受信したメール」を`SINCE`検索で毎回取得し（IMAPは`BODY.PEEK[]`で取得して既読フラグを変更しない）、ADR 0003で確立済みの`t_transactions.source_hash`（Message-IDを一意識別子として含む）による`ON CONFLICT DO NOTHING`で重複を排除する。これにより、バッチの再実行や日をまたいだリトライに対して安全（冪等）になる。

## 4. カード下4桁の不一致は「取込拒否」ではなく「警告フラグ付きで取込」

ADR 0006の通り、口座解決の主軸は送信元アドレス×受信メールボックスであり、下4桁照合はあくまで補助的な整合性検証である。抽出した下4桁が登録済みの`card_last4`と一致しない場合でも取引の取込自体は継続し（実際に発生した支出データを欠落させないため）、`raw_data`に不一致フラグを記録した上でバッチサマリーの件数に計上し、手動補正画面での事後確認に委ねる。
