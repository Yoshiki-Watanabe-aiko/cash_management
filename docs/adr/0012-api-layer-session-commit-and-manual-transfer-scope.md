# バックエンドAPI層：セッションcommit責務の集約と手動振替紐づけの検証範囲

Phase 6（バックエンドAPI: ダッシュボード集計・取引管理・手動補正）の実装にあたり、以下の設計判断を確定した。

## 1. `Session.commit()`の責務を`app/db/session.py`の`get_db()`依存関係に集約する

Phase 0〜5のバッチ処理（`batch_orchestrator.py`等）はCLIエントリポイントが`SessionLocal()`を直接開き、ステップの節目で明示的に`session.commit()`していた。一方Phase 6で追加したFastAPIのAPIエンドポイントはHTTPリクエスト単位でトランザクション境界を切るのが自然であり、ルーターごとに`commit()`呼び出しを書かせると書き忘れのリスクがある。そこで`get_db()`を「正常終了時はcommit、例外発生時はrollbackしてから再送出、finally節で必ずclose」という形に変更し、`Depends(get_db)`を使う全てのAPIハンドラ・サービス関数はセッションへの変更を`flush()`までに留めてcommitを呼ばない方針とした。バッチ処理側の`SessionLocal`直接利用パターンは変更していない（CLIは複数ステップにまたがる1トランザクションを扱うため、リクエスト単位のcommitとは性質が異なる）。

## 2. 手動振替紐づけは要件定義書5.2章の条件3（摘要一致）を意図的に検証しない

自動検知（`transfer_detection.py`）は条件1(金額完全一致)・条件2(営業日0〜3日以内)・条件3(摘要への口座名義一致)の3条件を全て満たすペアのみリンクする。条件3を満たさない振替ペアは要件定義書5.2章の記述通り「手動補正画面でのリンクに委ねる」対象であるため、新設した`app/services/transfer_management.py`の`create_manual_transfer_link`は条件1・条件2のみを検証し、条件3(摘要マッチング)は検証しない。これにより、摘要に口座名義が現れない実際の振替（例: 「振込」とだけ表示される明細）もユーザーが目視で選んで手動リンクできる。誤操作防止のため、金額不一致・出金/入金の向きが逆・営業日ギャップ超過・既にリンク済みの取引の再利用は`TransferLinkError`で拒否する。

## 3. カテゴリ別支出（`category_breakdown`）は事業按分をせず取引全額で集計する

ダッシュボードの「今月の事業経費の進捗」（`budget_progress`）と「個人口座のキャッシュフロー」（`personal_cashflow`）は要件定義書5.4章・7章の記述通り`business_ratio`で按分した金額を使うが、「カテゴリ別の支出金額」ウィジェットは要件定義書7章に按分方式の指定がないため、取引金額の全額（振替除外のみ）をカテゴリごとに合計する方式とした。個人・事業を跨いだ「このカテゴリに全体でいくら使っているか」という直感的な内訳を優先した判断であり、按分後の内訳が必要になった場合は将来的にクエリパラメータで按分方式を切り替え可能にする拡張の余地を残している。

## 4. `recategorize_uncategorized`はカテゴリルールを事前に一括取得してから走査する

「未分類の取引にカテゴリルールを再適用」バッチ操作（要件定義書7章）は対象取引数が多くなり得るため、取引ごとに`m_category_rules`を再クエリするとN+1になる。`app/services/categorization.py`に`load_category_rules`（ルール一括取得）と`categorize_with_rules`（取得済みルールに対するマッチング）を追加し、`categorize`はその合成として再実装した上で、`recategorize_uncategorized`はルールを1回だけ取得してから全取引を走査する。
