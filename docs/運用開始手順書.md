# 運用開始手順書：統合資産・経費管理システム (cash_management)

最終更新: 2026-07-11

Phase 0〜10 の実装完了を受けて、実データでの運用を開始するための手順書。
セットアップコマンドの詳細はルート [`README.md`](../README.md) を、各設定値の意味は本書を参照する。

前提: Windows / PostgreSQL 18 インストール済み / Python 3.14 + uv / Node.js。

---

## Step 1. `.env` の設定

`backend/` で `.env.example` をコピーして `.env` を作成し、以下を編集する。

```bash
cd backend
cp .env.example .env
```

> **注意**: APIキー・パスワード類の実値はチャット（Claude Code含む）に貼り付けず、本人が直接 `.env` に転記する（`~/.claude/ENV_CONVENTIONS.md`）。

### 1-1. 必ず実値に書き換える項目

| 変数 | 設定する値 | 取得方法 |
|---|---|---|
| `DATABASE_URL` | `postgresql+psycopg://cash_mgmt_user:{パスワード}@localhost:5432/cash_management` | Step 2 でロール作成時に決めたパスワードを埋める |
| `DISCORD_WEBHOOK_URL` | バッチ通知先チャンネルのWebhook URL | Discordの通知先チャンネル → 歯車（チャンネルの編集）→ 連携サービス → ウェブフック → 新しいウェブフック → 「ウェブフックURLをコピー」 |
| `GMAIL_PERSONAL_ADDRESS` / `GMAIL_BUSINESS_ADDRESS` | カード速報メールを受信するGmailアドレス（個人用・事業用） | — |
| `GMAIL_PERSONAL_APP_PASSWORD` / `GMAIL_BUSINESS_APP_PASSWORD` | Googleアプリパスワード（16桁） | Googleアカウント管理 → セキュリティ → 2段階認証プロセスを有効化 → https://myaccount.google.com/apppasswords → アプリ名（例: cash_management）を入力して生成 |
| `ACCOUNT_HOLDER_NAMES` | 自分の口座名義の表記ゆれをカンマ区切りで列挙（例: `ヤマダタロウ,山田太郎,ヤマダ タロウ`） | 銀行明細・振込摘要に実際に現れる表記（カナ/漢字/スペース有無）を網羅する。振替自動検知の精度に直結する |
| `PG_DUMP_PATH` | `pg_dump.exe` のフルパス | 既定は `C:\Program Files\PostgreSQL\18\bin\pg_dump.exe`。インストール先が違う場合のみ変更 |
| `BACKUP_DIR` | バックアップ保存先の絶対パス | **Google Drive等のクラウド同期フォルダを推奨**（PC故障時にもバックアップが残る） |

### 1-2. デフォルトのままでよい項目（必要時のみ変更）

| 変数 | 既定値 | 備考 |
|---|---|---|
| `LONG_HOLIDAY_START` / `LONG_HOLIDAY_END` | 空 | 年末年始(12/31-1/3)はコード側に組込済み。GW等の長期休暇を追加したい場合のみ `MM-DD` 形式で設定 |
| `IMPORT_TRANSACTIONS_DIR` / `IMPORT_ASSETS_DIR` | `../import/transactions` / `../import/assets` | MFME CSVの手動配置先。**フォルダ自体は事前に作成しておく**（`backend/` からの相対パス＝プロジェクト直下 `import/`） |
| `CORS_ALLOW_ORIGINS` | `http://localhost:5173,http://localhost:3000` | Vite開発サーバーの既定ポートのままなら変更不要 |
| `BACKUP_DAILY_RETENTION_DAYS` / `BACKUP_MONTH_END_RETENTION_DAYS` | 30 / 365 | 保持日数を変えたい場合のみ |
| `BATCH_RETRY_COUNT` / `BATCH_RETRY_DELAY_SECONDS` | 3 / 5 | 機関別処理のリトライ設定 |

フロントエンド側は `frontend/.env.example` をコピーし、`VITE_API_BASE_URL` が既定（`http://localhost:8000`）のままでよいか確認する。

## Step 2. データベース準備（作成 → マイグレーション → 初期マスタ投入）

### 2-1. ロール・DB作成（未作成の場合のみ）

postgresスーパーユーザーで実行:

```sql
CREATE ROLE cash_mgmt_user WITH LOGIN PASSWORD '（Step 1 のDATABASE_URLと同じパスワード）';
CREATE DATABASE cash_management OWNER cash_mgmt_user;
```

### 2-2. マイグレーションとシード

```bash
cd backend
uv sync
uv run alembic upgrade head
uv run python -m app.db.seed
```

`app.db.seed` は冪等（何度実行しても重複しない）で、以下を自動投入する。**`m_institutions` を手動でINSERTする必要はない**。

- **金融機関 22社**: カード7社（三井住友・楽天・EPOS・PayPay・view・セゾン・イオン）、銀行9行（楽天・三井住友・三菱UFJ・ゆうちょ・PayPay・イオン・住信SBI・みずほ・りそな）、証券4社（楽天・SBI・三菱UFJeスマート・マネックス）、QR決済2種（PayPay・楽天Pay）
- **資産クラス 4種**: 現金・国内株式・投資信託・ローン
- **カテゴリ 21種**: 食費・交通費・消耗品費・事業収入 など

ローンの借入先が上記22社に含まれない場合のみ、psqlで追加する:

```sql
INSERT INTO m_institutions (institution_name, institution_type)
VALUES ('○○銀行', 'bank');  -- institution_type は bank/credit_card/securities/qr_payment のいずれか
```

なお `m_institutions.card_alert_sender_email`（カード速報メールの送信元アドレス）は**この時点では設定不要**。実メールサンプル入手後（課題#3）に設定する。

## Step 3. 口座（`m_accounts`）の登録 — マスタ管理画面から

口座はSQLではなく**口座管理画面から登録する**。バックエンド・フロントエンドを起動し、ブラウザで `http://localhost:5173/accounts` を開く。

```bash
# ターミナル1
cd backend && uv run uvicorn app.main:app --reload
# ターミナル2
cd frontend && npm run dev
```

### 3-1. 口座種別ごとの設定早見表

実際に保有している口座を「金融機関 × 個人/事業」の単位で1件ずつ登録する。

| 口座の種類 | account_type | 残高追跡<br>(tracks_balance) | 残高算出方式<br>(balance_method) | 初期残高<br>(opening_balance/date) |
|---|---|---|---|---|
| 銀行（MFME連携あり） | `bank` | ON | `moneyforward` | 不要 |
| 銀行（MFME連携なし） | `bank` | ON | `cumulative` | **必須**（下記参照） |
| 証券口座 | `securities` | ON | `moneyforward` | 不要 |
| クレジットカード | `credit_card` | **OFF** | 設定不可 | 不要 |
| チャージ式QR決済（PayPay残高等） | `qr_payment` | ON | `moneyforward`（MFME連携なしなら `cumulative`） | cumulativeの場合のみ必須 |
| 都度引き落とし式QR決済 | `qr_payment` | **OFF** | 設定不可 | 不要 |
| ローン | `loan` | ON | `moneyforward`（MFME連携なしなら `manual`） | 不要 |

- **balance_method の選び方**: マネーフォワードMEに連携済みの口座は `moneyforward`（資産CSV取込で残高スナップショットが作られる）。未連携の口座は `cumulative`（初期残高＋取引累積で毎日自動計算）。どちらも不可能な場合のみ `manual`。
- **opening_balance / opening_balance_date**（`cumulative` のみ必須）: **運用開始日を基準日**とし、その日の実残高を設定する。過去データは遡らない方針（課題#7）のため、ここが残高追跡の起点になる。

### 3-2. その他のフィールド

| フィールド | 設定内容 |
|---|---|
| `account_name` | 表示名。「楽天銀行 個人」「三井住友カード 事業」のように機関名＋用途で命名すると一覧で判別しやすい |
| `is_business` | 事業用口座なら ON |
| `default_business_ratio` | この口座の新規取引に適用される事業按分率(%)。**事業用口座=100 / 個人用口座=0 を推奨**（0より大きい値が「事業取引」として集計されるため、個人口座を100のままにすると全取引が事業扱いになる点に注意） |
| `moneyforward_account_name` | MFME資産CSVの「保有金融機関」列の表記と一致させる（未設定時は `account_name` で照合）。**正確な表記は実CSVを見てから調整すればよい**（課題#4の検証と同時に実施） |
| `card_last4` | クレジットカードのみ。カード番号下4桁（速報メールとの整合性検証用） |
| `is_active` | 通常は ON。解約・休眠口座のみ OFF |

## Step 4. Windowsタスクスケジューラ登録

管理者PowerShellで実行する（毎日深夜2時に日次バッチを起動。スリープ解除・時刻超過時の即時実行・ログオン時トリガー付き）:

```powershell
cd batch
powershell -ExecutionPolicy Bypass -File .\register_task_scheduler.ps1
```

`pg_dump` バックアップは日次バッチ内のステップとして実行されるため、バックアップ専用のタスク登録は不要。

## Step 5. 動作確認チェックリスト

1. **手動バッチ実行**: `cd backend && uv run python -m app.cli.run_daily_import` がエラーなく完走する
2. **Discord通知**: 機関ごとの成功/失敗サマリーが通知チャンネルに届く
3. **バックアップ**: `BACKUP_DIR` にダンプファイルが生成されている
4. **CSV取込**: マネーフォワードMEから明細CSV・資産CSVを手動エクスポートして `import/transactions/`・`import/assets/` に配置 → バッチ再実行 → 取引管理画面（`/transactions`）に取引が表示される
5. **残高スナップショット**: ダッシュボードの純資産推移に `cumulative`/`moneyforward` 口座の残高が反映される
6. **自動実行**: 翌朝、タスクスケジューラ経由の実行結果を Discord 通知または `t_batch_logs` で確認する（`Start-ScheduledTask -TaskName 'CashManagementDailyImport'` で即時テストも可）
